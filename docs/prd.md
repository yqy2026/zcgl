# 土地物业资产运营管理系统 PRD

## 1. 文档定位

本文档描述土地物业资产运营管理系统的产品目标态，是业务、产品、设计、研发和验收共同使用的产品需求基线。

本文档只回答“产品要做什么、为什么做、做到什么算完成”。字段、状态机、API 契约、代码实现证据、历史决策过程分别由其他文档承载。

|---|---|
| `docs/specs/domain-model.md` | 业务对象、字段、状态、枚举、统计口径 |
| `docs/specs/api-contract.md` | API、权限和请求契约 |

## 2. 背景与问题

土地/物业资产运营当前存在资产台账分散、合同链路割裂、证照与主体信息分散、统计口径不统一等问题。现实经营同时包含承租模式与代理模式，跨产权方、运营方、终端客户的业务关系复杂，依赖人工台账和线下口径容易造成漏记、重复统计和权限风险。

系统目标是建设统一资产运营平台，覆盖资产、项目、合同组、合同、台账、客户、分析和权限，支撑运营管理闭环。

## 3. 产品目标

### 3.1 MVP 一票否决目标

| 目标 | 验收含义 |
|---|---|
| 资产清晰可见 | 资产页和项目页可展示资产基本信息、租赁情况、客户摘要 |
| 合同组能力可用 | 承租模式与代理模式都可建模，且合同组内模式不混用 |
| 审核与纠错可审计 | 合同审核、反审核、作废、冲销、重建均有留痕 |
| 经营统计口径可用 | 支持总收入、自营租金收入、代理服务费收入、客户双指标 |

### 3.2 上线后三个月成功指标

| 指标 | 目标值 |
|---|---|
| 资产台账完整率 | >= 98% |
| 合同台账自动生成覆盖率 | >= 95% |
| 关键报表口径一致性问题 | <= 2 个/月 |
| 核心流程平均处理时长 | 下降 >= 40% |

| 范围 | 能力 |
|---|---|
| 资产管理 | 列表、详情、新增、编辑、删除/恢复、导入、附件 |
| 项目管理 | 项目与资产关系管理、项目汇总视图 |
| 租赁管理 | 承租模式、代理模式、合同组、合同生命周期、台账生成与维护 |
| 客户管理 | 客户增强信息、历史签约、风险标签、账期偏好 |
| 搜索能力 | 全局搜索入口与模块内搜索并存 |
| 权限体系 | 登录、会话刷新、角色权限配置、关键操作鉴权、基于主体绑定的数据范围自动注入 |
| 数据分析 | 概览、趋势、分布、收入拆分、客户双指标 |
| 文档处理 | PDF 上传、解析、批处理、进度追踪 |

### 4.2 Out of Scope

| 范围 | 说明 |
|---|---|
| 通用 BPM 审批流引擎 | MVP 不建设通用流程引擎 |
| 财务总账、税票、支付结算 | 只做运营台账和统计，不做财务总账系统 |
| SaaS 多租户产品化 | 不做跨法人 SaaS 多租户隔离产品化 |
| 高级 BI 建模 | 不做自定义分析语言和高级 BI 建模 |
| 用户侧物理删除 | 关键业务记录只允许逻辑删除、作废或冲销 |
| 产权证和权属方管理 | MVP 不纳入正式需求基线和功能验收 |

## 5. 角色、权限与数据范围

系统采用角色、绑定、视图三维正交模型。

| 维度 | 解决的问题 | 规则 |
|---|---|---|
| 角色 | 用户能做什么操作 | 基于 RBAC 与 ABAC 控制按钮和接口级动作 |
| 绑定 | 用户能看哪批数据 | 基于用户绑定的产权方或运营方主体自动过滤 |
| 视图 | 同一数据用什么经营口径解读 | 仅分析和大屏使用 owner / manager 口径 |

核心规则：

- 用户权限 = 角色可执行动作与主体绑定数据范围的交集。
- 多角色权限默认取并集。
- 权限管理员只可管理用户、角色、权限配置，不可查看业务数据。
- 多绑定用户在常规列表中展示数据并集，并按业务实体主键去重。
- 分析和大屏不能混合产权方口径与运营方口径，必须按视图口径展示。
- 管理员和审计类用户可按授权查看全量数据。

## 6. 核心业务口径

### 6.1 客户定义

客户不是固定身份，而是合同对方主体。同一主体可在不同合同中扮演不同角色。

客户列表以主体为唯一行，通过角色标签展示其在合同中的角色。历史合同保留签署时主体名称快照，不因主档后续变更而追溯覆盖。

### 6.2 经营模式

系统支持两种经营模式：

| 模式 | 结构 | 收入口径 |
|---|---|---|
| 承租模式 | 运营方从产权方承租，再向终端客户转租 | 自营租金收入 |
| 代理模式 | 产权方与终端客户直租，运营方提供代理管理 | 代理服务费收入 |

合同组只能对应一种经营模式，不允许在同一合同组内混用承租和代理。

### 6.3 合同关系分类

| 分类 | 所属模式 | 含义 |
|---|---|---|
| 上游 | 承租模式 | 产权方与运营方之间的内部入约合同 |
| 下游 | 承租模式 | 运营方与终端客户之间的对外出约合同 |
| 委托 | 代理模式 | 产权方与运营方之间的委托管理协议 |
| 直租 | 代理模式 | 产权方与终端客户之间的对外出约合同 |

### 6.4 到期后新签

MVP 不提供“续签”业务能力。合同到期后如继续合作，按新签流程重新创建合同组和合同，历史合同保持到期只读。

## 7. 功能需求

### 7.1 资产域

| 编号 | 需求 | 验收标准 |
|---|---|---|
| REQ-AST-001 | 资产主数据统一管理 | 支持分页、筛选、排序、搜索、软删除、恢复、导入、模板校验 |
| REQ-AST-002 | 资产与项目、权属关系可追踪 | 单资产同一时点只有一个当前有效项目和主产权主体，关系变更可追溯 |
| REQ-AST-003 | 关键主数据支持审核与反审核 | 审核状态可追踪，关键字段审核后受控，反审核后可重新提审 |
| REQ-AST-004 | 资产详情口径清晰展示 | 展示资产基本信息、租赁情况、客户摘要，按四类合同关系汇总 |

### 7.2 项目与主体关系域

| 编号 | 需求 | 验收标准 |
|---|---|---|
| REQ-PRJ-001 | 项目关系遵循运营管理语义 | 项目绑定运营管理方，通过资产间接关联产权方 |
| REQ-PRJ-002 | 项目详情默认按当前有效资产汇总 | 项目默认只聚合当前有效资产，不混入历史失效资产 |

### 7.3 租赁与合同组域

| 编号 | 需求 | 验收标准 |
|---|---|---|
| REQ-RNT-001 | 合同组作为主业务对象 | 合同组承载一笔经营关系，包含模式、主体、资产、合同、结算和风险信息 |
| REQ-RNT-002 | 承租模式与代理模式并行支持 | 两种经营模式都可建模，同组不混用，一个资产不关联多个有效合同组 |
| REQ-RNT-003 | 合同生命周期与状态流转 | 合同支持草稿、待审、生效、已到期、已终止等状态，合同组状态派生只读 |
| REQ-RNT-004 | 关键合同联审 | 金额、周期、计费、分润、主体、资产等关键变更触发联审 |
| REQ-RNT-005 | 反审核与纠错闭环 | 已出账数据不允许直接反审核，必须作废或冲销后重建 |
| REQ-RNT-006 | 台账自动化 | 合同生效后自动生成台账，支持查询、批量更新、导出、补偿和服务费台账 |

### 7.4 客户域

| 编号 | 需求 | 验收标准 |
|---|---|---|
| REQ-CUS-001 | 客户增强信息管理 | 展示基础信息、历史签约、风险标签和账期偏好 |
| REQ-CUS-002 | 客户统计双指标 | 同时展示客户主体数和客户合同数，并支持合同关系分类拆分 |

### 7.5 搜索域

| 编号 | 需求 | 验收标准 |
|---|---|---|
| REQ-SCH-001 | 全局搜索入口与对象范围 | MVP 搜索覆盖资产、项目、合同组、合同、客户 |
| REQ-SCH-002 | 搜索结果组织与排序 | 支持全部视图和分组视图，按相关度和业务置顶排序 |
| REQ-SCH-003 | 搜索权限过滤 | 未授权对象不返回，搜索结果受主体绑定数据范围约束 |

### 7.6 认证与授权域

| 编号 | 需求 | 验收标准 |
|---|---|---|
| REQ-AUTH-001 | 会话安全与统一鉴权 | 登录、刷新、退出闭环，关键写操作必须鉴权 |
| REQ-AUTH-002 | 数据范围上下文自动注入 | 查询、统计、搜索按用户主体绑定过滤，分析按视图口径隔离 |
| REQ-AUTH-003 | ABAC 数据策略引擎 | 支持策略包、角色绑定、拒绝优先、缓存失效和日志模式 |

| REQ-PTY-002 | Party 数据来源与创建路径 | 支持初始化导入和业务过程创建，合同提审前主体必须已审核 |
| REQ-APR-001 | 资产审批流第一阶段 | 支持资产审批实例、待办、我发起、时间线、通过、驳回、撤回和通知 |

| 编号 | 需求 | 验收标准 |

## 8. 非功能需求

| 分类 | 要求 |
|---|---|
| 安全 | PII 字段必须加密，状态变更请求必须防 CSRF，关键操作必须审计 |
| 权限 | 所有业务查询必须按主体绑定过滤，未授权对象不可见 |
| 性能 | 常规列表 P95 <= 800ms，统计和搜索 P95 <= 2s |
| 并发 | 核心实体使用乐观锁，冲突返回明确错误 |
| 幂等 | 批量台账更新、补偿任务、审批回调等关键写操作必须幂等 |
| 可维护性 | API 统一版本化，路由层和业务层分离 |
| 兼容性 | MVP 支持 Chrome 最新两个大版本和 Edge 最新版 |

## 9. 验收标准

MVP 放量前必须同时满足：

| 门槛 | 标准 |
|---|---|
| G1 | 资产页和项目页能清晰展示资产信息、四类合同关系租赁情况和客户摘要 |
| G2 | 合同组能力可用，支持承租、代理、关键合同联审和状态流转 |
| G3 | 经营统计口径可用，支持总收入、自营租金、代理服务费、客户主体数、客户合同数 |

建议验收覆盖：

- 创建资产重名拦截。
- 面积不一致拦截。
- 资产已关联合同时删除拦截。
- 资产详情按四类合同关系展示租赁情况。
- 项目必须绑定运营管理方。
- 多绑定用户常规列表展示并集去重数据。
- 合同组模式校验阻止承租和代理混用。
- 合同审批通过后自动生成台账。
- 合同纠错后旧台账作废、新台账重建。
- 客户双指标正确去重展示。
- PDF 抽取失败后可重试和人工修正。
- 搜索结果不返回未授权对象。
