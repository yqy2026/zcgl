## 一、结论摘要

该计划文档结构完整、覆盖面广，但存在多处 **门禁命令语义错误** 与 **文本/命令口径不一致**，会导致：

1. 本应阻断的问题被误放行（假通过）；
2. 实际已满足条件仍被阻断（假失败）；
3. 团队在跨环境执行时出现不可复现结果。

在修复下述 P0/P1 项之前，不建议将本文档作为“唯一执行门禁基线”。

## 二、问题清单（分级）

## P0（阻断级）

### P0-01：多处 grep 门禁未启用扩展正则，`|` 与 `?` 语义失效（高风险假通过）
- 证据：L1173, L1252, L1254, L1431, L1446
- 现象：命令使用 `grep -rElw "organization_id|ownership_ids?|..."` 与 `"ownership_ids?"`，但未加 `-E`。
- 影响：
  - `|` 被当作普通字符（非“或”）；
  - `?` 被当作普通字符（非“可选”）；
  - 零引用门禁大概率失真，可能直接放过残留字段。
- 建议：统一改为 `grep -rE...`（或 BRE 写法 `\|`、`\?`），并在文档固定一套正则规范。

### P0-02：P3d Exit 的 feature flag 验签命令与“三选一”语义冲突（逻辑自相矛盾）
- 证据：文本语义 L875-L884（A/B/C 三选一）；命令 L1231
- 现象：命令末尾强制 `test -f docs/release/evidence/capability-guard-env.md && ...`，即使已满足 A 或 B，仍要求 C 文件存在。
- 影响：
  - 与文档“三选一”描述冲突；
  - 在已走 A/B 路径时产生假失败。
- 建议：拆分为“声明来源三选一”与“必填证据项”两层门禁，或明确改写文本为“C 文件必需”。

### P0-03：P3e 收口规则存在硬冲突（“全部移除” vs “两文件保留”）
- 证据：L1095（移除所有 `types/*.ts` deprecated 旧字段） vs L1264（`organization.ts`/`ownership.ts` 明确保留到 Phase 4）
- 影响：执行者按不同段落理解会给出相反动作，触发评审争议与返工。
- 建议：将 L1095 改为“移除所有旧字段，**不含明确例外文件**（见 L1264）”。

## P1（高优先级）

### P1-01：核心门禁依赖文件当前仓库不存在，缺少“创建即通过模板”落地步骤
- 证据：L1195-L1213, L1484-L1490；仓库实况：
  - `docs/plans/execution/phase3d-authz-freeze.md` 不存在
  - `docs/release/evidence/capability-guard-env.md` 不存在
  - `frontend/tests/e2e/auth/authz-minimal.spec.ts` 不存在
- 影响：P3d Entry/Exit 在执行初期必然阻断；若无模板，团队会临时造文档导致口径漂移。
- 建议：在计划中增加“Day-0 初始化步骤”与最小模板内容（建议附复制即用骨架）。

### P1-02：旧字段门禁对注释/字符串高度敏感，未提供统一“代码位点过滤”策略
- 证据：L1424（提到会命中注释）但主命令仍为纯 grep 文本扫描
- 影响：迁移注释、错误提示文案、mock 文本都可能触发假失败。
- 建议：
  - 新增“语义扫描优先、文本扫描兜底”策略；
  - 至少将注释与字符串常量处理规则写成固定流程（例如先清理注释，再跑门禁）。

### P1-03：P3d 关键行为门禁主要依赖 grep，行为正确性验证比重偏低
- 证据：L1214-L1234（大量 grep 门禁）
- 影响：字符串存在不等于运行时生效，尤其是路由守卫链与 refresh 防风暴。
- 建议：提升行为测试权重：把 `@authz-minimal` 与关键集成用例放到“必须先于 grep 通过”的前置位置。

### P1-04：PowerShell 与 Bash 门禁语义并不完全对齐
- 证据：L1451+（PowerShell 段）与 L1418+（Bash 段）
- 现象：Bash 有 `PHASE3_GREP_EXCLUDES` 统一排除；PowerShell 多处命令未体现等价排除参数复用。
- 影响：Windows 与 Linux 执行结果可能不一致。
- 建议：补充 PowerShell 版本的统一排除变量接入示例（含 `Select-String` 路径过滤）。

### P1-05：P3d Entry 中 `constants/routes` 审计范围巨大但缺少结果格式约束
- 证据：L1210
- 影响：不同人会给出不同粒度审计结论，难复核、难追责。
- 建议：规定审计输出结构（文件、保留/迁移/删除、责任人、截止日期、验证命令）。

## P2（改进级）

### P2-01：章节内部“建议/必须”混用较多，执行优先级认知成本高
- 证据：全篇多处 `推荐/建议/必须/硬门禁` 混杂
- 影响：执行现场仍需二次解释。
- 建议：统一标签体系：`BLOCKER` / `REQUIRED` / `RECOMMENDED`。

### P2-02：部分命令链过长，可维护性偏低
- 证据：L1208, L1231（长链 `&&`）
- 影响：定位失败原因成本高。
- 建议：拆分为脚本化步骤（每步单一断言），并输出明确失败信息。

### P2-03：回滚段落缺少“回滚后观测窗口”要求
- 证据：L1578-L1589
- 影响：回滚后是否真正恢复稳定缺少统一口径。
- 建议：补充 30~60 分钟关键指标观察（403、5xx、关键页面成功率）。
