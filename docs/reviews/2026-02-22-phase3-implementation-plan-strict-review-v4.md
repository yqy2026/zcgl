# Phase 3 实施计划严格审阅报告（v4）

- 审阅方式：全文逐段 + 门禁命令可执行性核查 + 仓库现状交叉验证（frontend/backend/docs）

## 一、阻断级问题（P0，需先修）

### P0-1：P3d Entry 的 seed 门禁命令存在 Bash 语法错误（会直接执行失败）
- 定位：计划文档 [L1209]
- 现状：命令使用 `((./.venv/bin/alembic history ...))` 包裹子命令；在 Bash 中 `(( ))` 是**算术上下文**，不是命令分组。
- 影响：P3d Entry 关键门禁无法执行，阻断阶段启动。
- 建议修复：改为命令分组 `((cmd)) -> (cmd)`，或显式 `if command -v rg ...; then ...; else ...; fi`。

### P0-2：D9 Step 1 的 Bash 门禁命令同样误用 `(( ))`
- 定位：计划文档 [L1628]
- 现状：`((test -f ... && grep ... ) || grep ... || (...))` 同样进入算术解析，语法/语义均错误。
- 影响：D9 验签第一步无法稳定执行，P3d Exit 判定失真。
- 建议修复：改为 `(( ))`→`( )`，或拆成明确的 `if/elif/else` 三分支断言。

## 二、高风险问题（P1）

### P1-1：P3d Exit 把 E2E 作为第一证据，但仓库当前缺少指定强制用例文件
- 定位：计划文档 [L1230] 要求 `@authz-minimal` 为强门禁
- 仓库现状：`frontend/tests/e2e/auth/authz-minimal.spec.ts` 当前不存在。
- 风险：执行到 Exit 时才发现前置缺失，导致阶段尾部阻塞。
- 建议：在 P3d Day-0 明确“先创建最小 E2E 用例骨架”作为前置任务。

### P1-2：P3d Entry 依赖冻结/证据文档，当前仓库默认不存在，执行顺序易误触阻断
- 定位：计划文档 [L1200], [L1214], [L1280]
- 仓库现状：`docs/plans/execution/phase3d-authz-freeze.md`、`docs/release/evidence/capability-guard-env.md` 均不存在。
- 风险：若执行者从 §5.1 直接跑 Entry 门禁，会在未跑 §5.2.1 时失败。
- 建议：把“先执行 §5.2.1”升级为 P3d Entry 的第 0 条硬前置（并在表格中显式写“先后顺序”）。

### P1-3：D9 验签强依赖生产证据，放在 P3d Exit 会把开发阶段与发布阶段强耦合
- 定位：计划文档 [L882] + D9 Exit 门禁区间
- 风险：开发联调环境无法满足 release-id / startup-log / protected-route-check 的生产证据要求，造成“开发完成但门禁不通过”。
- 建议：将 D9 分为「开发完成门禁（代码+测试）」与「发布门禁（证据链）」两层，不在同一 Exit 强绑定。

### P1-4：部分门禁仍为文本 grep，存在“注释命中/常量折叠”误判空间
- 定位：计划文档 [L1231] 及 §6.2 多处 grep 门禁
- 风险：可能出现假阳性（注释、文案）或假阴性（间接常量、动态拼接）。
- 建议：关键门禁（动作词、resource 名、路由守卫元数据）优先 AST 脚本化，grep 仅用于快速巡检。

### P1-5：D9 Step 2 检查条件过弱，无法证明“flag 已接入主生效链”
- 定位：§6.3.1 Step 2（`grep ... ENABLE_CAPABILITY_GUARD|capability guard`）
- 风险：只要有注释或无效引用即可通过，不保证 `App.tsx` / `AppRoutes.tsx` 真正接线。
- 建议：替换为 AST/精准文件断言（例如强制检查 `App.tsx` 出现 `import.meta.env.VITE_ENABLE_CAPABILITY_GUARD` 且参与守卫分支）。

### P1-6：旧字段零引用门禁与“中文提示文案”清理绑定过紧，执行成本高且易反复
- 定位：§6.2（中英文旧字段与中文列名 grep 全清）
- 风险：非运行时代码（说明文案/测试注释）也会触发阻断，导致多轮非功能性返工。
- 建议：阻断门禁仅覆盖运行时入口与请求载荷；文案层面改为“审计项”而非阻断项。

### P1-7：计划体量与历史噪音过大，执行者难以快速定位“当前应执行版本”
- 定位：全文（1831 行 + 长版本历史）
- 风险：跨人接手时容易错用过期口径、漏掉最新冻结条款。
- 建议：拆分为「执行版（短）」+「历史版（长）」两份文档；执行版仅保留当前有效门禁与命令。
