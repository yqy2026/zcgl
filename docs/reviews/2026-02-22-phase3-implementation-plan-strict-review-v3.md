**审阅日期**：2026-02-22  
**审阅方式**：全文条款审阅 + 仓库实况交叉核验（脚本/目录/命令依赖/关键代码现状）

## 1. 总结（结论先行）

当前计划文档覆盖面很高，但仍存在**执行阻断与可操作性风险**：

- **P0（阻断）2 条**：按文档执行会直接卡死或首轮门禁失败。
- **P1（高风险）7 条**：会显著增加“通过文档检查但落地失败/争议”的概率。
- **P2（中风险）5 条**：主要影响维护成本和跨人协作稳定性。

结论：**建议先修复 P0 + 关键 P1，再启动正式子阶段执行。**

## 2. P0 阻断问题（必须先处理）

### P0-1：关键门禁硬依赖 `rg`，但环境未声明/未满足

- 证据（计划）：`§5.1 P3d Entry` 使用 `... | rg -n "20260219_phase2_seed_data_policy_packages|..."`（约 L1208）。
- 证据（实测）：工作区环境 `command -v rg` 返回 `rg_missing`。
- 影响：P3d Entry 门禁在不少环境会直接失败（并非业务失败，而是工具缺失失败）。
- 建议：
  1) 在计划中补“前置依赖检查”；
  2) 给出 `grep -E` 等价回退命令；
  3) 或改为项目脚本封装，避免人手拼命令。

### P0-2：P3d 强制证据文件/目录为硬门禁，但仓库未提供初始骨架

- 证据（计划）：`§5.1 P3d Entry` 要求 `docs/plans/execution/phase3d-authz-freeze.md` 与 `docs/release/evidence/capability-guard-env.md` 存在并具备模板节（约 L1200-L1218）。
- 证据（实测）：当前仓库存在 `docs/plans/`、`docs/reviews/`，但**无** `docs/plans/execution/`、**无** `docs/release/` 目录。
- 影响：团队首次执行会在 Entry 阶段因“文件/目录不存在”失败，且失败与业务无关。
- 建议：在计划中新增 Day-0 脚手架步骤（`mkdir -p` + 模板文件）并提供最小模板内容。

## 3. P1 高风险问题（强烈建议开工前收敛）

### P1-1：D9 验签命令过长且耦合过高，定位失败原因困难

- 证据：`§5.1 P3d Exit` 的 D9 命令为多层 `((...||...||...) && ... && ... && !grep ...)`（约 L1236）。
- 风险：一旦失败，执行者难以快速判断是“声明来源、代码引用、证据文件、反占位”哪一段失败。
- 建议：拆为 4 条独立断言（声明来源 / 代码开关引用 / 证据字段完整 / 反占位），并输出明确错误码。

### P1-2：`ROUTE_CONFIG`“显式废弃”与“字面量清理”仍有执行边界歧义

- 证据：`§4.4.1` 同时写“停止权限改造”与“允许最小字面量清理”（约 L829、L843）。
- 风险：不同执行者对“最小清理”范围理解不一致，容易产生争议性改动。
- 建议：加白名单化约束：仅允许改 `action/resource` 字面量，不允许改导出结构/类型定义。

### P1-3：多个关键门禁仍依赖文本 grep，存在语义误判空间

- 证据：P3d/P3e 多处门禁使用 grep 识别权限动作、资源名、旧字段残留（如 L1435 起）。
- 风险：注释、字符串常量、构造表达式都可能造成误判；“漏报+误报”并存。
- 建议：将关键规则（动作词、资源名、路由元数据）统一脚本化（AST），grep 仅作为辅证。

### P1-4：P3d Entry 要求“生产验签模板完整”，时点偏早，易诱发占位文档

- 证据：Entry 即要求 `capability-guard-env.md` 必填节齐全（约 L1218），而真实值通常在联调/发布后才生成。
- 风险：执行上容易先填占位内容，再在 Exit 强制反占位，增加返工。
- 建议：Entry 仅要求“模板存在”；完整性与反占位放到 Exit。

### P1-5：强门禁同时要求 Bash 与 PowerShell 等价，维护成本高且易漂移

- 证据：`§6.2/§6.3` 维护两套命令体系，且逻辑复杂。
- 风险：一侧命令修订后另一侧遗漏，出现“同规则不同结果”。
- 建议：统一收敛到 `node/python` 门禁脚本，再提供薄包装命令。

### P1-6：文档历史区过长（v1.11~v1.44 同文件），执行视图噪声高

- 证据：`§10 文档历史`占比很高（L1652 起）。
- 风险：实施者查当前口径成本高，易误读旧版本描述。
- 建议：主计划保留“当前生效规则 + 最近1~2版摘要”，其余转归档文档。

### P1-7：部分门禁对“人工验证证据”有要求，但缺统一产物规范

- 证据：`§5.1 P3c/P3d/P3e` 多处写“浏览器验证/联调记录”，未统一格式。
- 风险：验收证据粒度不一致，跨人复核主观性强。
- 建议：提供固定模板（字段：场景、账号、期望、结果、截图路径、日志编号、执行人）。

## 4. P2 中风险问题（建议阶段内清理）

1. `§6.2` 注释出现 `grep -rEl[w]` 记法，表达不直观，易误读。  
2. 多处命令默认“在仓库根目录执行”，未在每条命令前显式声明。  
3. 旧字段门禁与中文旧列名门禁分散在多个章节，阅读跳转频繁。  
4. 部分“建议/推荐”与“硬门禁”混排，执行优先级不够一眼清晰。  
5. 计划将大量质量判断压在人工浏览器验证，自动化占比仍可提高。
