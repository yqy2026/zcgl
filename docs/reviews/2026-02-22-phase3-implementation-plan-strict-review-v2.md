# Phase 3 实施计划严格审阅报告（v2）

**审阅对象**：`docs/plans/2026-02-20-phase3-implementation-plan.md`（v1.43）  
**审阅日期**：2026-02-22  
**审阅方式**：全文结构化审阅 + 与仓库现状交叉核验（命令/路径/脚本/目录）

---

## 1. 总体结论（先看）

该计划文档已经非常详尽，但仍存在若干会直接影响执行落地的硬问题：

- **P0（阻断级）问题 6 条**：若不修正，执行中会出现“按文档操作但门禁无法通过”或“误放行/误阻断”。
- **P1（高风险）问题 11 条**：会导致跨团队口径漂移、验收争议、排障成本大幅上升。
- **P2（中风险）问题 9 条**：主要是可维护性和执行效率问题，建议在 P3 开始前清理。

审阅判断：**不建议按当前版本直接执行**，建议先完成 P0 + 关键 P1 收敛后再进入正式实施。

## 2. P0 阻断级问题（必须先改）

### P0-1：`DB_PSQL_CMD` 默认值与仓库 Docker 服务名不一致（会直接命令失败）

- 现状：计划文档 §5.2 默认值为：`docker compose exec db psql -U zcgl_user -d zcgl`
- 仓库事实：`docker-compose.yml` 中服务名是 `postgres`，不是 `db`。
- 影响：所有依赖 `DB_PSQL_CMD` 的 Entry/Exit SQL 门禁在默认配置下直接失败。
- 建议修复：将默认值改为 `docker compose exec postgres psql -U zcgl_user -d zcgl`，并在文档加“若服务名自定义请覆盖环境变量”。

### P0-2：P3e “全量零引用”与门禁排除清单存在语义冲突

- 现状：§4.6 写“确认旧字段在前端全量零引用”；但 §5.2 排除了 `mocks`、测试目录、组织模块等。
- 影响：执行层会把“主运行链零引用”误当成“全量零引用”，造成交付宣称与事实不一致。
- 建议修复：统一术语为两层门禁：
  1) **运行时主链零引用（阻断）**；
  2) **仓库全量零引用（审计项）**。

### P0-3：部分门禁命令将“文件不存在”与“内容不命中”混为一类

- 现状：虽已有部分 `test -f`，但若维护者复制命令片段单独执行，仍容易出现“缺文件导致误判”。
- 影响：跨人执行时会产生“本地通过/他人失败”的不一致。
- 建议修复：所有关键门禁统一改为两段式：
  1) 显式存在性断言；
  2) 内容断言；
  并在每条命令前标注“不可拆分执行”。

### P0-4：P3d 验签规则仍偏“文档存在性导向”，缺少“运行时真实值”硬证据

- 现状：D9 双层规则强调 `capability-guard-env.md`，但“运行中的最终 flag 值”未形成强制命令化证据闭环。
- 影响：可能出现“文档写了 true，实际构建注入 false”的假通过。
- 建议修复：增加 Exit 必选证据：
  - 前端启动日志中的 `VITE_ENABLE_CAPABILITY_GUARD` 实际值；
  - 发布包环境变量快照（脱敏）；
  - 一条受保护路由实测截图/日志（flag on/off 各一）。

### P0-5：P3d 门禁对 `ROUTE_CONFIG` 的策略与门禁 A 的扫描目标存在执行张力

- 现状：D6 冻结“`ROUTE_CONFIG` 显式废弃，不再继续改造”；但门禁 A 仍扫描 `constants/routes.ts` 中非标准动作词。
- 影响：执行者无法判断“该文件是否允许为过门禁而改字面量”，容易引发争议。
- 建议修复：明确二选一策略：
  - A) `ROUTE_CONFIG` 仅保留定义，不再参与任何动作词门禁；
  - B) 允许最小字面量清理，但禁止新增运行时引用。

### P0-6：P3e 删除 `usePermission` 的“物理删除”要求与遗留 Router 链路耦合风险仍偏高

- 现状：文档有 A/B 处置，但默认删除路径仍依赖大量 grep 人工判读。
- 影响：容易在最后阶段引入编译断裂（尤其是隐式 re-export 或类型引用）。
- 建议修复：在 P3e Entry 增加“删除前 dry-run 任务”：
  - 先将 `usePermission.tsx` 改名并执行 `pnpm type-check`，零错误后再物理删除。

## 3. P1 高风险问题（建议开工前收敛）

### P1-1：文档体量过大且历史版本过密，执行信息与历史噪声耦合

- §10 历史跨度极长，实施人员很难快速抽取“当前唯一有效口径”。
- 建议：把“执行版”与“历史版”拆分：
  - 主计划仅保留当前有效规则；
  - 历史迁移到归档文档。

### P1-2：同一目标存在“冻结决策门禁 + 实现门禁 + 证据门禁”三层叠加，操作成本高

- 例如 P3d 对 D7/D8/D9 同时要求冻结文档、代码落地、发布证据。
- 建议：给每条规则定义“唯一权威门禁”（其余改为说明），避免重复阻断。

### P1-3：大量 grep 门禁易受注释/字符串/示例代码干扰

- 虽有说明，但仍大量依赖文本扫描。
- 建议：关键门禁（动作词、路由元数据）优先 AST/TS 脚本，grep 降级为辅证。

### P1-4：`rg` 被用于关键 Alembic 门禁，但文档未声明依赖前置

- 建议在 §6 前置环境中加“需安装 `ripgrep`，否则改用 `grep -E` 等价命令”。

### P1-5：P3c “组织页只读”策略虽明确，但“服务层保护拦截”的验收不够可测试化

- 建议补充明确断言：写接口调用必须返回固定错误码/文案，且 UI 不仅是 disabled。

### P1-6：P3b 对后端契约依赖很多，但“超时未交付时的降级决策模板”仍不足

- 建议新增统一模板：阻断项、临时降级、风险窗口、回收时间、责任人。

### P1-7：P3d 的 refresh 防风暴规范强，但缺“统一埋点字段字典”

- 建议把埋点字段（名称、类型、枚举）单独冻结，避免前后端观测口径漂移。

### P1-8：P3e 对 `tenant_party_id` 收紧有量化门槛，但缺“样本抽检规则”

- 建议补充抽样策略（按项目/合同类型/时间段），避免只看总量掩盖局部风险。

### P1-9：计划中“主路由真源”强调 `AppRoutes`，但缺“禁止新建旁路守卫”的静态规则

- 建议新增 lint 规则或脚本：禁止在非 `App.tsx` 主链新增权限判定入口。

### P1-10：多处命令默认 Bash 数组语法，跨环境复制时失败概率高

- 建议给每条关键命令同时给出“可直接单行复制版”。

### P1-11：回滚定义偏前端，缺“后端兼容窗口有效性”持续校验

- 建议在回滚观测 30~60 分钟内增加一次后端兼容抽检任务（不是仅回滚前抽检）。

## 4. P2 中风险问题（建议阶段内逐步清理）

1. 部分章节术语混用：`零引用`、`主链零引用`、`运行时零引用`。  
2. 章节对同一规则重复叙述，阅读负担大。  
3. 伪代码与真实实现差异风险仍存在（虽已改进）。  
4. 验证计划中“人工验证项”较多，缺标准化记录模板。  
5. “文件影响总览”统计值缺自动化生成来源。  
6. 多处命令将 `frontend/src` 扫描范围写法不统一。  
7. 对 `mocks` 的处理口径分散在多处章节，不利于执行一致。  
8. 部分门禁命令较长，缺“失败示例”说明，排障成本高。  
9. 文档更新频次高，但无“当前执行基线哈希/版本锁定”机制。

## 6. 最小整改顺序（建议按此执行）

## 7. 审阅结论

该计划在覆盖面和前置风险意识上明显优于常规实施文档，但当前版本仍有“少量硬阻断 + 多处高复杂门禁设计”的问题。若先完成上述 P0 修复，再进入 P3 实施，可显著降低“文档正确、执行失败”的概率。
