# Phase 3 实施计划（v1.31）严格审阅报告（补充轮）

**审阅日期**: 2026-02-22  
**目标文档**: `docs/plans/2026-02-20-phase3-implementation-plan.md`  
**审阅目标**: 在既有评审基础上，继续挖掘“可执行性、门禁可靠性、跨平台一致性、发布安全”层面的剩余问题。

---

## 1. 总结结论

本轮发现 **10 项问题**（P0: 3，P1: 4，P2: 3）。

其中最关键结论：**文档中的多条 grep 门禁命令混用了 PCRE 语法（`\s` / `\b` / `[\s\S]`）与 `grep -E`，存在“误判通过/误判失败”风险**，会直接削弱 P3d/P3e 的硬门禁可信度。

---

## 2. 关键问题清单

## P0（阻断级）

### P0-1：D10 互斥门禁正则不可可靠执行（高概率失真）
- **证据**：L1175
- **现状**：
	- 命令使用 `grep -rEn` + 正则 `adminOnly\s*... [\s\S]{0,200} ... permissions`。
	- `grep -E` 并不等价支持 PCRE 语义：`\s`、`[\s\S]` 在不同实现下行为不一致，且该表达本质依赖跨行匹配能力。
- **风险**：D10（`adminOnly` 与 `permissions` 互斥）门禁可能被绕过，导致配置错误未被拦截。
- **建议修复**：
	1. 改为 AST/TS 脚本校验（首选）。
	2. 若必须 grep，改为两步：先单独定位包含 `adminOnly` 的路由块，再在同块内查 `permissions`，禁止依赖跨行 ERE 正则。

### P0-2：旧字段“零引用”门禁大量使用 `\b`，在 `grep -E` 下存在边界语义风险
- **证据**：L1139, L1199, L1201, L1353-L1368
- **现状**：多条关键门禁使用 `grep -rEl "\borganization_id\b|..."`。
- **风险**：不同 grep 实现中 `\b` 并非统一“单词边界”语义，可能出现误报/漏报，直接影响“P3e 收口完成”判断。
- **建议修复**：
	- 统一改为 `grep -w`（单词匹配）+ 分字段独立扫描；
	- 或改用 `ripgrep --pcre2` 并在文档显式声明工具依赖。

### P0-3：生产 flag 验签策略与示例命令存在“默认关停”发布风险
- **证据**：L862（D9 强制显式声明） vs L1386（缺失时写入 `VITE_ENABLE_CAPABILITY_GUARD=false`）
- **现状**：示例命令会在缺失 `.env.production` 时直接写入 `false`。
- **风险**：在“复制命令即执行”的流程中，可能把生产守卫默认写成关闭，且仍满足“显式声明”字面条件。
- **建议修复**：
	- 将自动创建改为**硬失败提示**（缺失即阻断，不自动写默认值）；
	- CI 增加 `VITE_ENABLE_CAPABILITY_GUARD` 允许值白名单和环境级强校验。

## P1（高优先级）

### P1-1：章节顺序错误，降低执行可读性
- **证据**：L1377 为 `6.4`，L1421 才出现 `6.3`。
- **风险**：执行者按编号查阅时容易漏执行浏览器冒烟矩阵。
- **建议**：交换两节顺序或重编号。

### P1-2：P3d Exit 的 `AuthGuard` 检查仍偏“文件存在性”，未对真实生效链做强约束
- **证据**：L1173-L1174
- **现状**：门禁强调“不在 AppRoutes 引用”，但未验证“其他入口（如延迟注册/动态装配）”是否重新接回主链。
- **建议**：新增运行时路由快照校验（构建后 route tree snapshot），把“主链唯一真源”为可执行断言。

### P1-3：门禁命令跨平台依赖未完全收敛
- **证据**：L1139-L1208（Bash 命令群）与 L1377-L1416（PowerShell 子集）
- **现状**：关键门禁的 PowerShell 等价覆盖不完整，仍有大量 Bash-only 断言。
- **风险**：Windows 团队执行口径不一致，评审证据不可比。
- **建议**：为 P3d/P3e 的每条“硬门禁”补齐 PowerShell 等价命令。

### P1-4：部分门禁仍是“grep 结构证据”，缺少行为级自动验证
- **证据**：L1165, L1172, L1175 等
- **现状**：多处靠 grep 判定“已接入守卫/已互斥”。
- **风险**：可通过注释、死代码或常量定义误导门禁。
- **建议**：将 E2E `@authz-minimal` 升格为 P3d 核心门禁主证据，grep 仅作为辅证定位。

## P2（优化级）

### P2-1：`request_id` 日志要求未定义来源契约
- **证据**：L1054-L1062（静默 refresh 诊断日志）
- **风险**：前端无法稳定拿到 request_id 时，日志规范难落地。
- **建议**：补充“request_id 来源优先级”（响应头 > 错误体 > 本地生成）。

### P2-2：`PartySelector` 的降级策略有 UX 文案，但缺少埋点口径
- **证据**：L480-L490（三级阈值降级）
- **建议**：新增埋点字段（`result_count`, `is_truncated`, `search_query_length`）以支持后续容量决策。

### P2-3：文档历史极长且包含大量“执行细节”，主计划可读性下降
- **证据**：L1457 之后版本历史长段
- **建议**：保留近 5 版摘要，其余沉淀到单独归档文档并链接引用。

---

## 3. 建议整改顺序（最小阻断路径）

1. **先修 P0-1/P0-2**：替换所有关键 grep 正则（`\s` / `\b` / `[\s\S]`）为可靠实现。  
2. **再修 P0-3**：去掉“自动写入 `false`”命令，改为缺失即失败。  
3. **修 P1-3**：补齐 PowerShell 等价门禁，统一跨平台证据。  
4. **最后处理 P1/P2 结构与可维护性项**（编号、历史归档、埋点补充）。

---

## 4. 审阅结论

当前 v1.31 在“架构思路完整性”上已明显成熟，但在“门禁命令可执行性与可验证性”上仍有关键短板。  
若不先修复本报告的 P0 项，Phase 3 的“硬门禁”将存在形式化通过、实质未达标的风险。
