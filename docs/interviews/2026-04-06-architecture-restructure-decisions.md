# 架构重构决策采访记录：三维正交模型与业务规则确认

- **日期**：2026-04-06
- **主题**：根因分析后的架构决策 —— 角色/绑定/视图三维正交模型确立、合同组规则确认、安全策略分期、API 契约变更
- **背景**：在审读需求规格说明书与三个活跃补丁方案（视角残留根除、RBAC 角色重组、业务规格更新）后，发现 `perspective` 一词同时承载角色(Role)、绑定(Binding)、视图(View) 三个正交概念，导致方案之间概念纠缠、越陷越深。经根因分析后，就 6 个关键业务决策点向业务方确认。

---

## 根因诊断摘要

**核心病灶**：`perspective`（视角）一词同时承载了三个本应正交的概念维度：

| 维度 | 含义 | 应有边界 |
|------|------|----------|
| **角色 (Role)** | 你能做什么操作？ | 纯 RBAC 权限矩阵 |
| **绑定 (Binding)** | 你属于哪个组织，看哪批数据？ | 纯 Party 过滤 |
| **视图 (View)** | 同一数据用什么业务口径解读？ | 纯展示/统计逻辑 |

三个方案各自的盲区：
- **方案 1**（视角残留根除）：纯符号重命名，`DataScopeContext` 仍混淆绑定和视图
- **方案 2**（RBAC 角色重组）：ABAC 策略 condition 中绑定 Party 过滤，把角色和数据范围重新纠缠
- **方案 3**（业务规格更新）：移除唯一结构性约束（模式互斥）却无替代方案

需求文档中发现 8 个隐藏矛盾（详见对话上下文 Part 1 §1.4 表格），需通过本次决策逐一消除。

## 问答记录

### Q1：合同组模式 —— 保持互斥还是解绑？

- **问**：方案 3 要解除 `validate_revenue_mode_compatibility()` 的经营模式校验锁，§15 访谈冻结写"合同组单模式不混用"，§4.1/REQ-RNT-001 又写"系统放开约束不强制拦截"。这三个说法互相矛盾，以哪个为准？
- **答**：
  - **保持互斥**。合同组仍然单模式（承租或代理），不允许混合。
  - **不允许一个资产关联多个合同组**（这是新增澄清）。
  - 业务定义：合同组 = 一个上游合同 + 对应的一个或多个下游合同。
  - 不存在拼铺出租的情况。
  - 上游合同（产权方和运营方之间的）才有承租和代理的模式区别。
  - **结论**：方案 3 的"解除经营模式校验锁"必须撤回，`validate_revenue_mode_compatibility()` 保留。

### Q2：双绑定用户大屏如何展示？

- **问**：§8.3 写"全局视角切换入口清晰可见，且支持手动切换"，§5.2 写"不需要用户手动选择或切换"。大屏/统计模块对双绑定用户应如何展示？
- **答**：
  - **单绑定用户**：隐藏 Tab 组件，直接展示对应口径。
  - **双绑定用户**：大屏顶部中间优雅出现 `[ 产权方口径 | 运营方口径 ]` Segment 切换器。
  - 默认激活其主要角色的那一侧。
  - **结论**：§8.3 第一条删除，替换为大屏双绑定口径选择的具体 UX 描述。

### Q3：Deny-Overrides / SoD —— MVP 还是 vNext？

- **问**：§5.1 要求"Deny Overrides"和"职责分离（SoD）"，但代码完全未实现。MVP 是否必须？
- **答**：**vNext**。MVP 阶段不要求 Deny-Overrides 和 SoD，标记为 vNext 候选需求。

### Q4：ABAC 激活 —— MVP 还是 vNext？

- **问**：ABAC 策略模板已就绪但 `abac_role_policies` 从未填充，当前所有授权 fallback 到 RBAC。MVP 是否需要激活 ABAC？
- **答**：**MVP 必须**。ABAC 是安全底线，MVP 需要激活。

### Q5：X-Perspective header 处置

- **问**：当前前端通过 `X-Perspective` HTTP header 传递视角信息给后端，这与"不需要用户手动选择"的设计有冲突。如何处置？
- **答**：
  - **废弃 `X-Perspective` header**。
  - 改为 query parameter `?view_mode=owner` 或 `?view_mode=manager`。
  - `view_mode` 仅用于分析/大屏端点，指定统计口径视图；常规 CRUD 端点不使用。
  - **结论**：在需求文档中标注 `X-Perspective` 废弃计划，改为 `view_mode` query param。

### Q6：perm_admin（权限管理员）数据可见范围

- **问**：权限管理员能否看到业务数据（资产、合同等），还是只能管理用户/角色/权限？
- **答**：**完全隔离**。`perm_admin` 只能管理用户、角色、权限配置，不能看业务数据。

## 决策对现有方案的影响

| 决策 | 影响 |
|------|------|
| Q1（互斥保持） | 方案 3 P1 "解除经营模式校验锁"撤回，`validate_revenue_mode_compatibility()` 保留 |
| Q2（大屏 Tab） | 方案 1 需扩展范围，不仅是重命名，还需支持 ViewMode 概念 |
| Q3（Deny/SoD vNext） | 方案 2 Phase 2 中 ABAC deny 修复仍需做，但 Deny-Overrides 策略框架推迟 |
| Q4（ABAC MVP） | 方案 2 Phase 2 的 ABAC 激活优先级提升，必须在 MVP 内完成 |
| Q5（废弃 header） | 方案 1 的 `X-Perspective` 引用全部需要重写为 `view_mode` query param |
| Q6（perm_admin 隔离） | 方案 2 Phase 1 角色定义中 `perm_admin` 需明确标注零业务数据权限 |
