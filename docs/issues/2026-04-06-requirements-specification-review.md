# `requirements-specification.md` 审阅意见

## 状态
- 日期：2026-04-06
- 审阅对象：`docs/requirements-specification.md`
- 关联附录：`docs/features/requirements-appendix-fields.md`、`docs/features/requirements-appendix-modules.md`
- 当前状态：历史审阅，已被 `docs/prd.md`、`docs/specs/`、`docs/traceability/` 文档体系重构吸收。
- 原始结论：旧综合文档已具备 SSOT 骨架，但还不适合作为“冻结后可直接开发/测试”的最终版本；建议先修正文档，再继续扩展实现与验收。

> 注：本文保留 2026-04-06 当时的审阅上下文，旧文档路径为历史引用；当前需求入口以 `docs/prd.md`、`docs/specs/domain-model.md`、`docs/specs/api-contract.md` 和 `docs/traceability/requirements-trace.md` 为准。

## 总体判断

本次审阅发现 5 类高风险问题：

1. 范围未闭环：产权证、权属方、用户/角色/主体绑定管理等核心模块在范围、搜索、数据实体中出现，但没有对应功能需求。
2. 术语与状态漂移：旧合同分类、旧实体名、`X-Perspective` 过渡契约、审核状态定义在多个章节和附录之间不完全一致。
3. 关键业务口径未冻结：多资产合同的合同级记账与资产视图展示口径、实收/收缴率、自动风险标签、ABAC 最小策略集等尚未定义到可开发、可测试的程度。
4. 状态机耦合过深：资产审核、合同审核、纠错链路、审批域第一阶段之间缺少统一状态映射，存在实现歧义。
5. 非功能需求偏粗：性能、安全、兼容性、并发/幂等、容灾与审计留存要求不够具体，难以作为验收依据。

## 其他问题

1. 主文档、字段附录、模块附录之间已有明显漂移，建议每次需求修订后做一次“术语表 + 状态表 + 模块清单”联动校对。
2. 追踪矩阵已经夹带部分实现细节和过渡契约，建议保持“需求目标态”表达，不要把迁移期历史包袱直接写进长期基线。
3. 当前最小验收场景过少，且偏实现校验，建议按“正常流程 + 异常流程 + 权限边界 + 数据修正 + 统计口径”补成可执行验收集。

## 建议的修订优先级

### P0：先修再继续开发

1. 补齐或明确移除：产权证、权属方、用户/角色/主体绑定管理需求。
2. 统一关键术语：合同四分类、合同/台账实体名、审核状态值、`X-Perspective` 目标态。
3. 增补三张表：权限矩阵、合同状态机、审批状态映射表。
4. 明确多资产合同金额口径：若保持合同级记账不分摊，需写清资产视图展示和汇总约束；若未来要分摊，再单独定义规则。
5. 冻结分析口径：实收、收缴率、服务费、客户统计四分类。

### P1：进入开发前应补齐

1. 完善 PDF 异常流程与人工修正闭环。
2. 补充自动风险标签规则或下调范围。
3. 扩展最小验收场景，覆盖双绑定、多角色、纠错、续签、批量导入失败等边界。
4. 补足非功能需求的并发、幂等、兼容性和备份恢复要求。

### P2：文档治理优化

1. 更新主文档版本日期。
2. 清理附录与正文的旧术语残留。
3. 将“现状/过渡/目标态”分层表达，避免 SSOT 与迁移计划混写。
